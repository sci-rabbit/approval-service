# DESIGN

Короткое описание устройства сервиса: модель данных, границы, обработка повторов,
события и известные компромиссы.

## Границы сервиса

Сервис отвечает только за процесс согласования. Публикации, сценарии, материалы,
пользователи и workspace живут в других сервисах и приходят сюда как **строковые
идентификаторы** (`source_id`, `reviewer_user_ids`, `workspace_id`, `actor_user_id`).
Сервис их не валидирует против внешних систем и ничего о них не хранит, кроме id.

Слои (сверху вниз):

- `api/` — HTTP-роуты, зависимости FastAPI (в т.ч. auth);
- `services/` — бизнес-логика и правила переходов;
- `repositories/` — доступ к БД (тонкая обёртка над SQLAlchemy);
- `models/` — ORM-модели; `schemas/` — Pydantic (вход/выход); `dto/` — датаклассы между слоями;
- `enums/`, `exceptions/`, `core/` — общие типы, ошибки, конфиг/БД/auth.

## Модель данных

```
approval_requests (id UUID PK)
  ├─ workspace_id, source_type, source_id, title, description
  ├─ status: pending | approved | rejected | cancelled
  ├─ created_by, created_at
  ├─ decision_by, decision_comment, decided_at
  ├─< approval_reviewers  (reviewer_user_id)          FK → CASCADE
  └─< approval_events     (event_type, actor_user_id, message, created_at)  FK → CASCADE

idempotency_keys (key PK) → workspace_id, request_id, created_at
outbox_events (id PK) → aggregate_id, event_type, payload JSON, created_at, published_at
```

- **approval_requests** — агрегат заявки. Индексы: `workspace_id`, составной
  `(workspace_id, status)` под выборки списка по workspace.
- **approval_reviewers** — кто может принимать решение по заявке.
- **approval_events** — аудит-след: `created`, `approved`, `rejected`, `cancelled`.
  Каждое изменение оставляет запись «кто и что сделал».
- **idempotency_keys** — дедупликация создания (см. ниже).
- **outbox_events** — транзакционный outbox для будущей публикации событий.

Итоговое решение денормализовано в саму заявку (`status`, `decision_by`,
`decision_comment`, `decided_at`) — чтобы читать состояние без обхода событий,
при этом полная история остаётся в `approval_events`.

## Изоляция workspace

Данные одного workspace недоступны из другого на двух уровнях:

1. **Auth**: `require(...)` сверяет `X-Workspace-Id` с `{workspace_id}` из пути;
   несовпадение → 403.
2. **Запросы**: каждый SELECT фильтруется по `workspace_id`. Запрос чужой заявки
   по прямому id из другого workspace возвращает 404 (факт существования не утекает).

## Обработка повторов (идемпотентность)

Создание заявки требует заголовок `Idempotency-Key`. Ключ пишется в
`idempotency_keys` вместе с id созданной заявки. При повторе того же ключа сервис
не создаёт дубль, а возвращает ранее созданную заявку. Так «retry» клиента (сеть,
таймаут) безопасен. Решения (approve/reject/cancel) идемпотентны по своей природе —
см. правила переходов.

## Правила переходов состояний

- Решение принимает только пользователь из `reviewer_user_ids` заявки (иначе 403),
  и только при наличии действия `approval:decide` / `approval:cancel`.
- Заявка меняет статус только из `pending`. После любого финального статуса
  (`approved` / `rejected` / `cancelled`) повторное решение → **409** — заявка не
  может перейти в другое финальное состояние.

## События и интеграция

Используется паттерн **transactional outbox**: при каждом успешном изменении в той же
транзакции пишется запись в `outbox_events` (создание, approve, reject, cancel).
Событие и изменение данных коммитятся атомарно, поэтому «изменение без события» или
наоборот невозможно. Отдельный публикатор (в брокер/шину) пока не реализован —
`published_at` зарезервирован под него. Это готовая точка расширения для интеграции
с другими сервисами через события.

## Безопасность данных в ответах/логах/событиях

В публичные ответы, логи и события попадают только id и бизнес-поля. Секретов, токенов,
email, storage keys, signed URLs и сырых provider-payload сервис не принимает и не хранит,
поэтому утечь им неоткуда. Payload события содержит только `requestId`, `workspaceId`,
`status`, `actorUserId`. Ответы описаны явными Pydantic-схемами (whitelist полей),
а не отдают ORM-модель целиком.

## Известные компромиссы

- **Auth — заглушка.** Identity берётся из заголовков без проверки подписи/срока;
  клиент сам заявляет свои действия. Годится для локального запуска; в проде это
  делает gateway. Замена локализована в `core/auth.py`.
- **Outbox без публикатора.** События копятся в таблице, но никуда не отправляются —
  нет воркера-релея и брокера.
- **Список без истории событий.** `GET .../approval-requests` отдаёт лёгкую схему
  (без `events`), чтобы не тянуть десятки событий на заявку; события видны в карточке
  одной заявки. Пагинация — простые `limit/offset` (limit ≤ 50), без курсора.
- **Миграции и enum-типы.** На Postgres `downgrade` явно удаляет enum-типы
  (`create_table` их создаёт, но `drop_table` не удаляет) — иначе повторный upgrade
  падал бы на «type already exists».
- **Нет доменной валидации внешних id** — `source_id`, `reviewer_user_ids` и т.п.
  принимаются как есть, существование во внешних сервисах не проверяется.
